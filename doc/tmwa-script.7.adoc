== Name

tmwa-script - the TMWA server-side scripting language

== Synopsis

TMWA script source is plain text loaded by the map server (see
*tmwa-map*(1)).  Most scripts define NPCs, but the same language and the
same bytecode interpreter are used in several other contexts (see
*DESCRIPTION*).  A trivial NPC script looks like:

----
example.gat,30,40,0|script|Example NPC|123,{
    mes "Hello, world.";
    close;
}
----

== Description

TMWA has a single embedded scripting language.  Script source is compiled
to bytecode at server startup and the same interpreter
(`run_script` / `run_script_l`, see _src/map/script-call.cpp_) executes
that bytecode in every context below.  The language is therefore *not*
NPC-specific, even though scripted NPCs are by far the most common use.

A running script always has an *origin* (the NPC, item or other object the
bytecode belongs to) and may have an attached player, the *RID* (the
"running ID", the player a builtin like *mes* or *getitem* acts on).  Some
contexts attach a player automatically; others run with no player attached
and a builtin that needs one will fail unless the script first calls
*attachrid*.

=== Execution contexts

The interpreter is entered from the following places.  All of them run the
same language; the differences are only in what triggers the script and
whether a player is attached.

NPC dialogue and interaction::
A player clicking a scripted NPC runs that NPC's script body
(`npc_click`).  *next*, *menu*, *input* and similar builtins suspend the
script; the player's later replies resume it (`npc_scriptcont`).  The
clicking player is the RID.

NPC event labels::
Labels named `On<Something>` inside an NPC script are entry points
triggered by the server rather than by a click.  *donpcevent* and the
`callfunc`-style cross-NPC calls invoke them explicitly; the server
invokes them implicitly:
+
* `OnInit` runs for every NPC once, at server start.
* `OnClock<hhmm>`, `OnDay<...>`, `OnHour<...>`, `OnMinute<...>` and
  related labels run on a wall-clock schedule.
* `OnTimer<ms>` labels run on per-NPC timers
  (*initnpctimer*, *startnpctimer*).
* `OnTouch` runs when a player steps onto the NPC's touch area.
* `OnPCLoginEvent`, `OnPCLogoutEvent`, `OnPCDieEvent`,
  `OnPCKillEvent` and `OnMobKillEvent` run on the corresponding player
  or mob event; these are broadcast to every NPC defining the label.

Mob "death" / kill events::
A monster can be spawned with an associated NPC event (for example by
*monster* or *areamonster*).  When the monster dies the server runs that
event as an ordinary NPC event-label script, with the killing player as
RID.

Player-triggered timers::
*addtimer* schedules an NPC event to run for a specific player after a
delay; it executes as a normal event-label script when it fires.

Item scripts::
Items carry script fragments compiled by the same parser.  A *use*-type
item runs its `use_script` when consumed; equipment runs its
`equip_script` when equipped (and on recalculation of stats).  The
player using or wearing the item is the RID.  See _src/map/pc.cpp_.

The "magic" / spell system::
TMWA no longer has a separate spell language in the shipped server.  A
spell is just an NPC event registered with the *registercmd* builtin: a
chat message a player types is matched against the registered
invocations and, on a match, the associated NPC event runs as a normal
event-label script (`magic_message`, see _src/map/npc.cpp_).  The
historical standalone spell interpreter survives only under
_old-doc/old-spell-language_ and is not part of the running server.

User-defined functions and subroutines::
*callfunc* and *callsub* call, respectively, a named `function` script
object or a label in the current script.  These are not a separate
context: they push a return frame and keep running on the same
interpreter and the same RID.

NOTE: Atcommands (the `@`/`#` GM commands) are *not* a script context.
They are implemented in C++ in _src/map/atcommand.cpp_.  Scripts interact
with them only indirectly, for example through the *remotecmd* builtin.

=== Variable scopes

Variable names carry their scope in a sigil prefix; a trailing `$`
marks a string variable.

[cols="1,4"]
|===
| _name_ | NPC-scope variable, shared by the script's NPC
| `@`_name_ | temporary variable, attached to the RID for the current run
| `$`_name_ | global variable, shared server-wide and persisted
| `.`_name_ | function-scope (local) variable
| `.@`_name_ | function-scope temporary variable
|===

Names without a sigil that match a known player parameter (for example
`Hp`, `Zeny`, `BaseLevel`) read and write that parameter on the RID.

== Builtin functions

Each entry below lists the builtin's name and a signature derived from the
argument-signature string in the `builtin_functions[]` table of
_src/map/script-fun.cpp_.

In a signature, an argument shown as a type name is a placeholder for a
value of that kind; arguments inside `[ ]` are optional; `...` marks a
repeatable trailing argument.  The argument-type vocabulary is:

[cols="1,4"]
|===
| `int`    | integer expression
| `str`    | string expression
| `expr`   | expression of any type
| `map`    | map name
| `x`, `y` | map coordinates
| `label`  | a label in the current script
| `func`   | a function or label reference
| `var`    | the name of a variable or player parameter
| `item`   | an item (by numeric id or name)
| `player` | a player (by name or account id)
| `event`  | an NPC event, written `NpcName::OnLabel`
| `timer`  | a timer interval, in milliseconds
| `mob`    | a monster species
| `status` | a status-condition (status-change) id
|===

A `-> type` suffix shows the value the builtin pushes back as an
expression result; builtins with no suffix are statements.
